For teams running it — and developers building it.

Run Claude Code agents on your own terms

Bandolier is a web app for running Claude Code agents as Kubernetes Jobs. Sign in with GitHub, point an agent at a repo or an issue, and Bandolier spins up an isolated pod that clones the code, runs the agent, and opens a pull request — while you watch live from a dashboard.

These docs serve two audiences. If you want to run and operate Bandolier for your own projects, start with the user tracks below. If you want to hack on Bandolier itself, jump to Developing Bandolier.

What is Bandolier?

You launch agents two ways:

  • From the dashboard — pick a repo, write a task (or select an open issue), choose a model, and deploy.
  • From a GitHub webhook — when an issue is opened in a configured repo, Bandolier automatically launches an agent to work on it and open a PR.
Your credentials, always. Every agent runs with the deploying user's own GitHub OAuth token, their model-provider credentials (AWS Bedrock, an Anthropic/OpenAI/Gemini API key, or a Claude Pro/Max · ChatGPT subscription login), and the cluster from their kubeconfig. There are no shared, server-side provider credentials — the available models are queried live from each user's provider.

How it works

Bandolier has two deployable pieces:

  1. The web app (src/) — a Next.js + tRPC application. It handles auth, stores per-user credentials, talks to the Kubernetes API, and serves the dashboard.
  2. The agent harness (agent-harness/) — a small Go binary baked into a container image alongside the Claude Code CLI, git, and gh. This runs inside each agent pod.

When you deploy an agent, the web app:

  1. Resolves your credentials (GitHub token, model provider, kubeconfig) from the database.
  2. Creates a namespace, a bandolier-agent ServiceAccount, and (optionally) a NetworkPolicy that isolates agent pods.
  3. Writes a short-lived per-job Secret holding only that run's credentials, owned by the Job so Kubernetes garbage-collects it when the Job is deleted.
  4. Creates a Kubernetes Job running the harness image, with the task wired in via environment variables.

The harness then clones the repo, runs claude --print, commits the work, pushes a branch, and opens a PR. The claude CLI speaks Anthropic and Bedrock natively; every other provider (OpenAI, Gemini, a subscription login, and more) is served through a gollm proxy the harness runs in-process, translating Claude Code's Anthropic-format traffic to the real backend. Logs stream back to the dashboard; if artifact storage is configured, the full transcript is uploaded to S3 so it outlives the Job's TTL.

Project setup

Six steps to a running local instance.

1 Install dependencies

pnpm install

2 Create a GitHub OAuth app

In GitHub → Settings → Developer settings → OAuth Apps → New OAuth App:

  • Homepage URL: http://localhost:3000
  • Authorization callback URL: http://localhost:3000/api/auth/callback/github

Copy the Client ID and generate a Client Secret. The app requests the repo and workflow scopes so agents can clone private repos and open PRs (including ones that touch .github/workflows/).

3 Configure environment

Copy the example file and fill it in:

cp .env.example .env

Minimum required for local development:

# A random secret — generate one with: openssl rand -base64 32
BETTER_AUTH_SECRET="…"

# From the GitHub OAuth app you just created
BETTER_AUTH_GITHUB_CLIENT_ID="…"
BETTER_AUTH_GITHUB_CLIENT_SECRET="…"

# Local Postgres (matches start-database.sh below)
DATABASE_URL="postgresql://postgres:password@localhost:5432/bandolier"
You also need a cluster for agents to deploy into: each user pastes their own kubeconfig in the app's settings, or a repo admin configures a shared kubeconfig for a repo in its settings.

4 Start Postgres & apply the schema

./start-database.sh     # starts a Postgres container from DATABASE_URL
pnpm db:push            # syncs the schema straight to your local database
db:push is for local development only. Self-host deploys apply the checked-in SQL migrations under drizzle/ (via the migrator image, run as a Helm hook). After changing src/server/db/schema.ts, run pnpm db:generate to emit a new migration and commit it.

5 Run the app

pnpm dev

Open http://localhost:3000 and sign in with GitHub.

6 Configure your account

Open Settings in the dashboard and add:

  • A model provider — Bedrock credentials, an API key, or a subscription login. For Claude Pro/Max, run claude setup-token locally and paste the sk-ant-oat01-… token; for ChatGPT, run codex login and paste ~/.codex/auth.json. API keys are validated before saving and the model picker is populated live; subscription credentials get a static model list and are verified on the first run.
  • A kubeconfig — the cluster your agents deploy into (a repo can also provide a shared one). Bandolier runs the Kubernetes client on its server, so exec-plugin kubeconfigs (the aws/gcloud credential plugins EKS/GKE emit) and ones that reference cert/key files on your laptop won't work — the server has neither those binaries nor those files. Generate a self-contained, token-based kubeconfig from your current cluster and paste that instead: 
# hosted — no checkout needed
curl -fsSL https://<your-host>/setup.sh | bash

# from a local checkout
./scripts/create-bandolier-kubeconfig.sh

# add --scoped for a least-privilege ClusterRole instead of cluster-admin
curl -fsSL https://<your-host>/setup.sh | bash -s -- --scoped
You can now select a repo, deploy an agent, and watch it work. 🎉
For users · Track 02

Workflow configuration

Optional pieces that shape how agents get triggered and constrained.

GitHub App & webhooks

To have agents launch automatically when issues are opened, install the Bandolier GitHub App.

  1. Create a GitHub App (Settings → Developer settings → GitHub Apps). Give it a webhook URL of https://<your-host>/api/webhooks/github, a webhook secret, and these repository permissions: Issues (read & write), Pull requests (read & write), Contents (read), Actions (read), and Metadata (read). Subscribe to the Issues, Issue comment, Pull request review comment, and Workflow run events. Issue comment and pull request review comment power resuming a run by commenting; Workflow run powers auto-resuming a run when a CI pipeline fails on the pull request it produced, for repos that opt in. Generate a private key.
  2. Set GITHUB_APP_ID, GITHUB_APP_PRIVATE_KEY (PEM, \n-escaped), and GITHUB_WEBHOOK_SECRET in the app's environment. Optionally set NEXT_PUBLIC_GITHUB_APP_SLUG so the repo-config UI links to the App's install page.
  3. Install the App on the repos you want Bandolier to act on. Events are delivered automatically — there's no per-repo webhook to add.

When an issue is opened, Bandolier verifies the App's signature, finds the Bandolier user linked to the GitHub account that opened it, and deploys an agent under that user's credentials — so clone/push and the resulting PR are attributed to the issue author. The "Bando picked up this issue…" comment is posted by the App itself (bandolier[bot]). Optionally, each repo can set a trigger phrase that issue text must contain.

Resuming a run by commenting

Commenting on an issue or pull request that a run already worked on resumes it: Bandolier finds the item's most recent run, spawns a follow-up under the commenter's credentials, and seeds it with the parent run's persisted transcript (when artifact storage is configured) so the agent picks up with full context. While the parent's PR is still open, the follow-up works directly on its branch; otherwise it starts fresh. Resumed tasks carry a "↻ resumed" chip in the dashboard.

This works for both a pull request's vanilla conversation comments and its inline review comments (left on a specific line of the diff). A review-comment resume additionally hands the agent the file, line range, and diff hunk the comment is anchored to, so it knows exactly which code the reviewer is pointing at.

Comments from bots are ignored (including Bando's own), a comment only ever resumes — one with no prior run does nothing — and a configured trigger phrase applies to comments too.

Repository system prompt

A repo admin can attach a system prompt to a repository in its settings — a blanket instruction (coding conventions, a review checklist, "always add tests") that Bandolier appends to the system prompt of every agent run for that repo: dashboard tasks, issue PRs, and webhook-triggered runs, across every provider and both one-shot and interactive sessions.

It's layered on top of Bandolier's own framing, never replacing it, so you don't need to repeat guidance in each task. Leave it blank for none.

Repository credentials & the maintainer gate

A repo admin can configure shared repo-level credentials (a kubeconfig and/or model-provider API keys) so everyone works on the same pooled infrastructure. Because those credentials are shared, executing on them is restricted to GitHub users with maintainer access or higher. A run only counts as using shared credentials when it actually resolves to the repo's kubeconfig or model key — a user with their own credentials is never gated.

  • From the dashboard / REST API: if your run would use the repo's shared credentials and you're not a maintainer, the deploy is rejected.
  • From a webhook (issue opened): if the issue opener isn't a maintainer, the run is held for approval. The bot comments asking a maintainer to approve — who can then react 👍 (or 🚀) to the bot's comment, or reply /bando approve (decline with /bando decline).
Approvals are one-shot and verified server-side — a non-maintainer's command or reaction is ignored. On approval the held run is dispatched exactly as it was originally built, attributed to the issue opener.

Agent isolation

Agent-pod network isolation is controlled by two environment variables:

VariableDefaultDescription
AGENT_NETWORK_POLICY true Apply a NetworkPolicy denying inbound and limiting egress to DNS + the public internet. Needs a policy-enforcing CNI (Calico/Cilium); a no-op under kindnet.
AGENT_EGRESS_BLOCKED_CIDRS RFC-1918 ranges Comma-separated CIDRs agents cannot reach (blocks lateral movement to in-cluster services).

A repo admin can loosen these per repository from the repo configuration modal. Both toggles are off by default: Allow in-cluster (private) egress drops the blocked-CIDR exclusion, and Allow all egress ports permits outbound TCP on any port instead of only 80/443.

Security: loosening egress weakens agent isolation. Agents run model-generated code with your credentials; in-cluster egress opens lateral movement and all-ports egress widens the exfiltration surface. Enable only for repos you trust, and turn the toggles back off when done.

Run artifacts

Each run's full transcript can be uploaded to S3 so it survives the Job's one-week TTL. Storage is configured per repository in the repo-config modal (admin-only, under Shared credentials → Run artifact storage): the repo names an S3 bucket it owns (AWS or any S3-compatible endpoint such as MinIO) plus scoped credentials. There is deliberately no server-wide bucket — run data belongs to the repo, never the operator.

This is independent of a run's structured output (the PR or issue URL), which is always persisted to the run row — so a finished run's output stays recoverable even when S3 isn't configured.

REST API

Besides the dashboard, agents can be listed and launched over a small REST API under /api/v1, authenticated with an API key (create one in the app; the token is shown once and only its hash is stored). 

# List tasks for a repo
curl -H "Authorization: Bearer bnd_…" \
  https://<your-host>/api/v1/repos/<owner>/<repo>/tasks

# List the models you can launch with (every provider you've configured)
curl -H "Authorization: Bearer bnd_…" \
  https://<your-host>/api/v1/repos/<owner>/<repo>/models

# Launch a task
curl -X POST -H "Authorization: Bearer bnd_…" -H "Content-Type: application/json" \
  -d '{"task":"Fix the flaky test in auth.spec.ts"}' \
  https://<your-host>/api/v1/repos/<owner>/<repo>/tasks

# Read one task
curl -H "Authorization: Bearer bnd_…" \
  https://<your-host>/api/v1/repos/<owner>/<repo>/tasks/<id>

# Rename a task
curl -X PATCH -H "Authorization: Bearer bnd_…" -H "Content-Type: application/json" \
  -d '{"displayName":"New name"}' \
  https://<your-host>/api/v1/repos/<owner>/<repo>/tasks/<id>

# Terminate a task
curl -X DELETE -H "Authorization: Bearer bnd_…" \
  https://<your-host>/api/v1/repos/<owner>/<repo>/tasks/<id>
MethodPathBodyPurpose
GET/api/v1/repos/{owner}/{repo}/models—List launchable models
GET/api/v1/repos/{owner}/{repo}/tasks—List tasks for a repo
POST/api/v1/repos/{owner}/{repo}/taskslaunch fields (below)Launch a task
GET/api/v1/repos/{owner}/{repo}/tasks/{id}—Read one task
PATCH/api/v1/repos/{owner}/{repo}/tasks/{id}{ "displayName": … }Rename a task
DELETE/api/v1/repos/{owner}/{repo}/tasks/{id}—Terminate a task

The models endpoint returns every model you can launch with, drawn from all the providers you've configured — the four first-class providers (Anthropic, Bedrock, OpenAI, Gemini) and every provider gollm supports (Groq, OpenRouter, Together, Mistral, DeepSeek, self-hosted vLLM/Ollama, and the rest of the ~90-provider catalog). Each entry carries the exact provider to pass as modelProvider — a gollm:<id> for the proxied ones.

The launch endpoint accepts everything the dashboard's deploy dialog can set, except interactive sessions. Every field except task/prompt is optional:

FieldTypeNotes

promptstringAlias for task.

modelstringA model id from one of your providers (see the models endpoint above).
modelProvideranthropic · bedrock · openai · gemini · gollm:<id>Pins which provider serves model. gollm:<id> (e.g. gollm:groq) selects a gollm-proxied provider — use the provider the models endpoint returns.

For users · Track 03

Self-hosting on Kubernetes

Own and operate the whole stack for your team.

Overview & images

The web app runs anywhere Node does — Vercel, a VM, or Kubernetes. To deploy the whole stack to a cluster, the repo ships a production image and a Helm chart. The web app is cluster-agnostic about where agents run (it uses each user's kubeconfig), so it can live on the same cluster as its agents or a separate one; its only hard dependency is Postgres.

ImageStagePurpose
ghcr.io/based64god/bandolierrunnerThe lean Next.js standalone server. This is the app.
ghcr.io/based64god/bandolier-migratemigratordrizzle-kit + the SQL under drizzle/. Runs migrations as a Helm hook.

The Helm chart (deploy/helm/bandolier) wires a Deployment, Service, Ingress, config/secret wiring, a pre-install/pre-upgrade migration Job, health probes (/api/health), a choice of database (external, a bundled StatefulSet, or an operator-managed CloudNativePG cluster), and optional in-cluster MinIO for run artifacts.

Helm quick start (bundled Postgres, port-forward)

Fastest way to see it running — no external database, no ingress:

helm install bandolier deploy/helm/bandolier \
  --namespace bandolier --create-namespace \
  --set postgres.mode=bundled \
  --set config.betterAuthUrl=http://localhost:3000 \
  --set secrets.betterAuthSecret="$(openssl rand -base64 32)" \
  --set secrets.githubClientId=<oauth-client-id> \
  --set secrets.githubClientSecret=<oauth-client-secret>

kubectl -n bandolier port-forward svc/bandolier 3000:80
# open http://localhost:3000
For the OAuth callback to work, set your GitHub OAuth app's callback URL to http://localhost:3000/api/auth/callback/github, and config.betterAuthUrl must match the URL you actually reach the app on.

Production (external Postgres + ingress + TLS)

Put your values in a file rather than on the command line so secrets don't land in shell history: 

# values.prod.yaml
image:
  tag: v0.1.0 # pin a release

config:
  betterAuthUrl: https://bandolier.example.com

ingress:
  enabled: true
  className: nginx
  host: bandolier.example.com
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
  tls:
    enabled: true
    secretName: bandolier-tls

secrets:
  betterAuthSecret: "<openssl rand -base64 32>"
  githubClientId: "<oauth-client-id>"
  githubClientSecret: "<oauth-client-secret>"
  databaseUrl: "postgresql://user:pass@your-db-host:5432/bandolier"
helm upgrade --install bandolier deploy/helm/bandolier \
  --namespace bandolier --create-namespace \
  -f values.prod.yaml
Keep real secrets out of Helm values. Committing secrets to a values file leaves them in Helm's release history. Manage the Secret out-of-band with kubectl create secret generic bandolier-env … and point the chart at it with secrets.create: false and secrets.existingSecret: bandolier-env. This composes with sealed-secrets / external-secrets operators.

Database options

postgres.mode selects how the app gets its database:

ModeWhat it deploysUse it for
external (default)Nothing — you provide secrets.databaseUrl.Production with a managed database.
bundledA single-replica StatefulSet + PVC.Evaluation / quick starts.
cnpgA CloudNativePG Cluster (HA, failover, backups).In-cluster production Postgres.

On every install/upgrade a hook Job runs drizzle-kit migrate against DATABASE_URL. Disable it with migrations.enabled=false if you apply the schema yourself. The app exposes GET /api/health — 200 when the process is up and Postgres is reachable, 503 otherwise — wired to both liveness and readiness probes.

One-click DigitalOcean

Don't have a cluster yet? deploy/terraform/digitalocean provisions the whole stack with OpenTofu — a DOKS cluster, managed Postgres, a Spaces bucket for run artifacts, optional DNS + Let's Encrypt TLS — and installs the chart, in one tofu apply. 

cd deploy/terraform/digitalocean
cp terraform.tfvars.example terraform.tfvars   # fill in your values
tofu init
tofu apply

The agent harness image

Agent Jobs run a built-in default image, ghcr.io/based64god/bandolier-agent-harness:latest (built and pushed by .github/workflows/agent-harness-image.yml). Pods always pull it (imagePullPolicy: Always).

To run a custom build, set a per-repo agent image override in the repo's settings. Because pods always pull, the image must be pushed to a registry the cluster can reach: 

docker build -t <your-registry>/bandolier-agent-harness:latest agent-harness
docker push <your-registry>/bandolier-agent-harness:latest
Private GHCR images are pulled automatically: Bandolier attaches a short-lived image-pull Secret (owned by the Job) authenticated with the triggering user's GitHub OAuth token — requiring the read:packages scope (requested at sign-in). For private images on other registries, configure the cluster's nodes with pull credentials yourself; Bandolier only brokers GHCR.

Security notes

  • Per-user credentials. Agents only ever run with the deploying (or issue-opening) user's own GitHub, model-provider, and cluster credentials. There is no server-side provider identity to fall back on.
  • Shared repo credentials require maintainer access. See the maintainer gate.
  • Per-job secrets. A run's credentials live in a Kubernetes Secret owned by its Job, deleted when the Job is. Finished Jobs are retained for one week.
  • Stored credentials are not encrypted at rest. User AWS/Anthropic keys and kubeconfigs are stored in Postgres in plaintext; API key tokens are stored only as SHA-256 hashes. Protect the database and serve the app over HTTPS.
  • Network isolation for agent pods is on by default but requires a policy-enforcing CNI to take effect.
For developers · Track 04

Developing Bandolier

Working on Bandolier's own codebase — not just running it.

Everything above is for running Bandolier. This track is for contributing to it: the Next.js web app under src/, the Go agent harness under agent-harness/, the embedded gollm proxy, and the Helm chart. The canonical reference is CONTRIBUTING.md.

Get the code running

Local development uses the same six steps as Project setup — pnpm install, a GitHub OAuth app, .env, ./start-database.sh, pnpm db:push, then pnpm dev. Two things matter specifically when you're changing code:

  • The /dev/* harness routes. The UI components render against fake data on inert /dev/* routes that contact no real services — no cluster, no model provider, no GitHub. That's what the browser smoke tests drive, and it's the fastest way to iterate on UI without wiring up credentials.
  • db:push vs migrations. pnpm db:push syncs the schema straight to your local database for development. After changing src/server/db/schema.ts, run pnpm db:generate to emit a new SQL migration under drizzle/ and commit it — that's what self-host deploys apply, and CI checks it exists.

The verification loop

Run all of these locally before you push — CI runs the same checks on every push and pull request (see .github/workflows/ci.yml): 

pnpm check      # ESLint + tsc --noEmit
pnpm test       # Vitest unit suite (once)
pnpm test:e2e   # Playwright browser smoke tests over the /dev/* routes

cd agent-harness && go test ./...   # the Go harness suite

pnpm check and pnpm test cover the web app; pnpm test:e2e boots a dev server and drives the UI harness routes; go test ./... (from agent-harness/) covers the Go binary that runs inside each agent pod, plus the vendored gollm module it embeds.

CI runs a few more suites that need a throwaway Postgres, so they're not in the loop above but will still gate your PR: the DB-backed integration tests (pnpm test:integration) and the product-flow browser specs (pnpm test:e2e:flow / pnpm test:e2e:authflow), plus Helm-chart linting and OpenTofu validation.

The wire-contract rule

wire-contract.json is the single source of truth for the constants that cross the TypeScript-server ↔ Go-harness process boundary (log markers, control sentinels, the effort allow-list). Because the two languages can't share a package, each side re-declares these constants in code and asserts them against the JSON:

  • TypeScript: src/lib/wire-contract.test.ts
  • Go: agent-harness/cmd/harness/wire_contract_test.go
If you change wire-contract.json, update the constants on both sides so both suites keep passing. Drift is caught in CI, not in production — keep it that way.

Two more vendored bits worth knowing: patches/kysely@0.29.2.patch is a pnpm patch (applied automatically on pnpm install) that re-exports migration constants drizzle-kit needs, and skills-lock.json pins the vendored neon-postgres skill to a specific upstream commit by content hash.

Test suites

Several suites cover the project. The unit and Go suites exercise the pure logic — fast, hermetic, and free of any database, network, or Kubernetes access; the integration suite runs real handlers against a throwaway Postgres; and the browser suites drive the UI in a real Chromium against inert harness routes.

SuiteCommandWhat it exercises
Web app unit pnpm test *.test.ts next to the code — parsing, validation, formatting, and crypto-token helpers.
Web app integration pnpm test:integration *.integration.test.ts — real tRPC procedures, REST/webhook handlers, and auth flow against a real, migrated Postgres.
Agent harness (Go) go test ./... The harness's pure helpers — slugging, branch naming, prompt building, PR-content parsing, provider detection.
Browser smoke pnpm test:e2e e2e/*.spec.ts drive UI components in Chromium against the /dev/* routes (no real services).
Product-flow browser pnpm test:e2e:flow / :authflow Fuller flows — the gated (shared-password) surface and the signed-in surface against a throwaway Postgres + fake Kubernetes.
The browser suites need a browser: install one first with pnpm exec playwright install chromium (add --with-deps on Linux). Set E2E_BASE_URL to reuse an already-running server instead of booting a fresh one.

FAQ

Frequently asked questions

Split the same way as the docs — users first, developers second.

For users

Running and operating a Bandolier instance.

Do I need to self-host to use Bandolier?

To use Bandolier you need access to a running instance — whoever operates it self-hosts it. If you're evaluating or standing one up yourself, follow Project setup for a local instance or Self-hosting for a shared one.

Whose credentials do my agents run with?

Always your own. Every agent runs with the deploying user's GitHub OAuth token, their model-provider credentials, and the cluster from their kubeconfig. There are no shared, server-side provider credentials — see Security notes.

Which model providers are supported?

AWS Bedrock, an Anthropic/OpenAI/Gemini API key, or a Claude Pro/Max · ChatGPT subscription login — each user supplies their own in Settings. The claude CLI speaks Anthropic and Bedrock natively; every other provider is served through the in-process gollm proxy.

How do I launch an agent without opening the dashboard?

Two ways: open a GitHub issue on a configured repo and Bandolier launches an agent automatically, or POST to the REST API with an API key.

My kubeconfig won't save, or agents won't deploy.

Bandolier runs the Kubernetes client on its server, so exec-plugin kubeconfigs (the aws/gcloud credential plugins EKS/GKE emit) and ones that reference cert/key files on your laptop won't work. Generate a self-contained, token-based kubeconfig with setup.sh or ./scripts/create-bandolier-kubeconfig.sh — see Configure your account.

Can I keep iterating on an agent's PR?

Yes. Commenting on the issue or pull request a run worked on resumes it under your credentials, seeded with the parent run's transcript. Inline review comments hand the agent the exact file, line range, and diff hunk.

Why is my webhook run "held for approval"?

Because it would use the repo's shared credentials and you're not a maintainer. Shared credentials are gated — a maintainer approves by reacting 👍 to the bot comment or replying /bando approve. See the maintainer gate.

For developers

Contributing to Bandolier's codebase.

What should I run before pushing?

The verification loop: pnpm check, pnpm test, pnpm test:e2e, and go test ./... from agent-harness/. CI runs the same checks plus the DB-backed and product-flow suites.

Do I need a cluster or a model provider to work on the UI?

No. The /dev/* harness routes render the UI components against fake data and contact no real services — no cluster, no provider, no GitHub. That's what the browser smoke tests drive. See Get the code running.

I changed wire-contract.json — what else must I update?

Both re-declarations of the constants: src/lib/wire-contract.test.ts (TypeScript) and agent-harness/cmd/harness/wire_contract_test.go (Go). CI catches drift. See the wire-contract rule.

I changed the schema — db:push or a migration?

pnpm db:push is for your local database only. To ship a schema change, run pnpm db:generate to emit a SQL migration under drizzle/ and commit it — that's what self-host deploys apply via the migrator image.

Which test suite covers what?

Unit (pnpm test) for pure helpers, integration (pnpm test:integration) for handlers against Postgres, Go (go test ./...) for the harness, and the browser suites for the UI. Full breakdown in Test suites.

What is gollm?

An embedded litellm-style LLM proxy and SDK written in Go, vendored under gollm/. The harness runs it in-process to serve every non-Anthropic/Bedrock provider, translating Claude Code's Anthropic-format traffic to the run's real backend.
