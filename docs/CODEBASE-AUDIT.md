## Scope & method

A 5×4 matrix of finder agents swept the repository — five areas (`src/server` backend, `src/app` frontend, `src/lib` + proxy shared libs, the vendored `gollm` Go gateway, and the `agent-harness` Go job runner + `e2e`) across four dimensions (test coverage, DRY/duplication, API design, general code smells). Each finder cited `file:line` evidence; each finding then went to a separate verifier agent told to disprove it and grep for counter-evidence. The three high-severity findings below were additionally re-verified by hand.

**Confidence caveats.** Findings are ranked `high` (active risk or serious drag on change), `medium` (real maintenance cost), `low` (opportunistic). The verification pass refuted 0 findings outright but down-graded several from high to medium, so treat `low` items as leads rather than settled defects. Severities reflect maintainability/coverage risk, not a security assessment (run `/security-review` for that). Generated files (drizzle migrations, `pricing_data.go`, `.next`) were excluded.

## High-severity findings

### H1. Owner-only mutation paths on the agents router (terminate, rename, sendInput, endSession, acpSend) are completely untested, and their shared primitive assertOwnsInteractiveJob is only ever mocked

- **Where:** `src/server/api/routers/agents.ts:480 + src/server/agents/authz.ts:39`
- **Area / dimension:** Server · Test coverage
- **Evidence:** Grepped `\.terminate(|\.rename(|\.sendInput(|\.endSession(|\.acpSend(` across all test files and e2e/: zero hits. authz.test.ts imports only assertRepoAccess and repoViewSelector (authz.test.ts:24-25); assertOwnsInteractiveJob (authz.ts:39-57, the selector `...,SPAWNED_BY_LABEL=...,bandolier.io/job=...` + NOT_FOUND-on-empty that gates all interactive-session writes) appears in tests only as a vi.fn stub (agents.getLogs.test.ts:19-26). terminate's tenant-isolation check — list via ownedSelector then match podName (agents.ts:499-511, comment: "never delete another user's pod by raw name on a shared namespace") — and rename's equivalent (agents.ts:363-375) are asserted nowhere. The read-path counterparts (getLogs/acpPull fallbacks) do have dedicated router tests, so the write side is the asymmetric gap: swapping ownedSelector for repoViewSelector in terminate would let any collaborator kill another user's pod with no failing test.
- **Fix:** Extend the agents.getLogs.test.ts harness (mocked k8s client + caller factory) with cases: terminate/rename reject NOT_FOUND when the pod list scoped by ownedSelector does not contain the target pod, and succeed when it does; sendInput/endSession/acpSend reject when assertOwnsInteractiveJob throws. Add direct unit tests for the real assertOwnsInteractiveJob (empty list → NOT_FOUND, selector contains SPAWNED_BY_LABEL and the job label).

### H2. The /api/gate password-verification endpoint (the seam that mints the gate cookie) has zero tests at any level

- **Where:** `src/app/api/gate/route.ts:6`
- **Area / dimension:** Shared libs · Test coverage
- **Evidence:** The POST handler verifies the shared password (`provided.length === password.length && timingSafeEqual(provided, password)`), mints the cookie via `gateToken`, sets it with `httpOnly/sameSite/secure/maxAge`, and routes the redirect through `safeFrom(fromField)`. `ls src/app/api/gate` shows only route.ts — no route.test.ts, unlike every other auth-relevant route (src/app/api/acp, agent-input, agent-runs, webhooks/github all have route.test.ts). Grep for "api/gate|gate/route" across all *.test.ts and e2e/*.mjs finds only proxy.test.ts:88, which merely asserts the path is gate-exempt. The helpers it composes (safeFrom, timingSafeEqual, gateToken in src/lib/gate.ts) are unit-tested in isolation (src/lib/gate.test.ts), but the glue is not: a refactor that drops the `safeFrom` call reintroduces an open redirect via the form's `from` field, and a lost `httpOnly`/`secure` cookie attribute or an inverted `ok` branch would ship with all suites green.
- **Fix:** Add src/app/api/gate/route.test.ts in the style of proxy.test.ts (real NextRequest with formData, vi.stubEnv for APP_PASSWORD/BETTER_AUTH_SECRET): correct password → 303 to `from` with GATE_COOKIE set and httpOnly/sameSite=lax/path=/ asserted; wrong password → 303 to /gate?error=1 with no cookie; `from=//evil.com` and `from=https://evil.com` → redirect to "/"; APP_PASSWORD unset → redirect to /; non-string form fields tolerated.

- **Where:** `agent-harness/cmd/harness/main.go:62`
- **Area / dimension:** Harness (Go) · Test coverage
- **Evidence:** go tool cover -func on the package (56.2% total) shows run 0.0%, resolveMode 0.0%, finish 0.0%, runAgent 0.0%, setupGit 0.0%, gitIdentity 0.0%; grep for resolveMode/finish/runAgent/setupGit across all *_test.go returns 0 hits. Untested risky branches: the resume guard `mode.prBranch != cfg.branch` (main.go:110) whose failure re-cuts an existing branch and aborts the run; the issue-mode loud-failure invariants (main.go:223-241: empty CLAUDE_TASK, empty AGENT_BRANCH, empty CLAUDE_SYSTEM_PROMPT must each error); and finish's ordering contract (main.go:329-357: rewriteCommitAuthors before latestCommitSubject, ensureCloses re-applied after generatePRContent rewrote the body, issue PRs keeping the issue title). The pieces (rewriteCommitAuthors, ensureDiffBase, ensureCloses, classifyPRCreate) are individually tested in main_test.go but no test asserts their composition or ordering, and fetchParentContext (transcript.go:137) is 0% too.
- **Fix:** Add unit tests for resolveMode's switch (pure config-in/mode-out for the repo/resume/issue-output branches; the three issue-mode error invariants need no gh). Then one integration test of run() in plain repo mode reusing the scaffolding that already exists in this suite: buildFakeClaude (acp_agent_test.go:60) on PATH, seedRemote (main_test.go:560) as origin, and a stub `gh` script — asserting clone, branch cut, authorship rewrite, and the openPR skip/create decision end to end.
